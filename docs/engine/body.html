The Body

Audio in → speech → a lip-synced 3D head out. Two completely different voice backends sit behind one function signature, hot-swappable mid-sentence. That swap is the engine.

This half lives in the embodied-dreamfinder repo: a browser client (Three.js avatar + Web Audio) and a small Express server that hands out tokens. Run it with npm start (serves on http://127.0.0.1:3001 - use the IP, not localhost, or Chrome blocks the mic) and, for the local-voice mode, npm run agent.

The crown jewel: one interface, two substrates

Both voice backends export the identical function:

export async function startVoicePipeline(room, localParticipant) { ... }

What makes them swappable is a fixed audio contract. Both substrates emit PCM16, 24 kHz mono, in 480-sample (20 ms) chunks, base64-encoded. Because the bytes downstream are identical, the avatar/playback layer cannot tell which backend produced them. Swap the producer, keep the consumer. That contract - not any class hierarchy - is the real interface.

The swap itself (triggered by a switch_voice_mode tool the model can call) tears down only the transport and resets the mouth; everything expensive survives:

Survives the swapTorn down
Web Audio context, the TalkingHead avatar, the Three.js scene, the spark particles, mood stateThe WebSocket / LiveKit transport, any in-flight audio, viseme morphs (reset to 0)

The avatar layer

Talking head + visemes

The avatar is a TalkingHead instance driving a GLB with 67 blend shapes (52 ARKit + 15 Oculus visemes). Lip sync runs in an AudioWorklet: the headaudio/ ML module classifies each audio chunk into a viseme name in ~10 ms and calls setViseme(name, value) to drive the morph targets. Mood is a string ('happy', 'thinking'...) mapped to blend-shape combos via setMood().

Sparks (mood-reactive particles)

sparks.js is a self-contained GLSL particle system: new SparkSystem(scene, opts) then setMood(), setEnergy(0..1) (driven by audio frequency bins), burst(). Palettes are swappable (warm gold for Dreamfinder, cool starlight for Lyra). Pure Three.js, no other deps - the most directly reusable file in the repo.

The papyrus QR conjure (River's signature move)

showQRCode(url) raises the right hand, conjures a 3D papyrus scroll bearing a QR code, tracks the hand bone each frame, and billboards to camera. The URL is hardcoded to imagineering.cc on purpose - never let an LLM choose the QR target, or you have a prompt-injection phishing vector. Tightly coupled to the TalkingHead armature; treat it as a reference for gesture+mesh choreography rather than a drop-in.

Embedding the avatar elsewhere - the fast path

You do not need the voice stack to use the face. avatar-renderer.js is a standalone, iframe-embeddable renderer with a tiny control surface:

// same-origin window functions:
__onAudioChunk(base64pcm)   // feed it PCM16 (24kHz, the contract above)
__setMood('happy')
__interruptPlayback()

// or cross-origin via postMessage:
iframe.contentWindow.postMessage({ type: 'audio', data: base64pcm }, '*');
iframe.contentWindow.postMessage({ type: 'mood',  mood: 'thinking' }, '*');

Feed it audio + mood from any source - your own TTS, a game server, a livestream - and it lip-syncs. No auth, no voice connection, just rendering.

How to build your own avatar on it

Step 1 - Bring your own GLB

Any rigged humanoid with compatible viseme blend shapes (15 Oculus or 52 ARKit) drops in. Load via the ?avatar= URL param or swap the default.

Step 2 - Choose how voice is produced

Reuse a substrate as-is, or write your own startVoicePipeline wrapping your STT/TTS/LLM. The only rule: emit audio on the PCM16 / 24 kHz / 480-sample / base64 contract and the avatar just works.

Step 3 - Drive mood and reskin

Map your character's emotions to mood strings, and swap the sparks palette (or the whole shader). Replace the papyrus conjure with your own gesture, or drop it.

Step 4 - Replace the persona

The system prompt and tool definitions are hardcoded constants in app.js (Dreamfinder vs Lyra). Swap them for your character. This is the most River-specific code in the body - expect to rewrite it wholesale.

How it reaches a brain

The body shares no code with the Dart brain. In local-voice mode the agent sends a line of text to a brain over HTTP and speaks the reply. That single text-in / text-out boundary is the seam between the two halves - covered on the roadmap page as the Brain API. It is why you can point this body at River's brain, your brain, or no brain at all (Realtime mode talks straight to OpenAI).
