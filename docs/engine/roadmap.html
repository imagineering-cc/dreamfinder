What would make it more engine-like

Andy's real question. The seams are already clean - but a thing becomes an engine when you depend on it and configure it, not when you fork it and edit it. Today Dreamfinder is mostly the second kind. Here is the short, concrete path to the first.

The throughline. Nothing below is a big rewrite. It is three extractions (pull clean code out into things you can depend on), two interfaces (turn a convention into a contract), and one rename (give the best-designed boundary in the system a name). All of it is already 80% true in the code; this is about making it declared rather than implied.

Brain: from "a folder of clean classes" to "a package you depend on"

Extraction 1 - publish the agent core as a Dart package

lib/src/agent/ (AgentLoop, ToolRegistry, ConversationHistory) is genuinely decoupled, but it lives inside the app. Pull it into a dreamfinder_engine package and have the app depend on it. Then "build your own bot" becomes dependencies: dreamfinder_engine: ^1.0.0 instead of "fork the repo and delete the parts you do not want." This single move is the difference between a codebase and an engine.

Interface 1 - make PlatformClient a real abstract class

Right now the platform seam is a convention: the main loop happens to call MatrixClient.sync() / .sendMessage() directly. Define an abstract class PlatformClient (sync, send, join, typing, isDm, isMentioned) and a Discord or Slack client becomes a compile-checked drop-in. The seam exists; it just is not yet a type the compiler enforces.

Builder - shrink main() from 1090 lines to ~30

The entry point mixes wiring, feature orchestration, and the sync loop. An engine would expose DreamfinderBot.builder() taking (platformClient, llmFn, tools, systemPromptBuilder) and running the loop, so an app's main() is a short, declarative assembly. The feature modules (kickstart, dream cycle, radar) become opt-in plugins discovered from config - the same pattern MCP servers already use.

Body: from "constants in app.js" to "character as data"

Extraction 2 - publish the renderer and sparks as npm packages

avatar-renderer.js (already has a clean window/postMessage API) and sparks.js (zero deps beyond Three.js) are near-publishable today. Ship them as @imagineering/avatar-renderer and @imagineering/sparks and other projects embed a talking head with an npm install.

Extraction 3 - lift the substrate's transport coupling

startVoicePipeline(room, localParticipant) takes a LiveKit room, so the substrate knows about LiveKit. The real interface is already the PCM contract (24 kHz / 480-sample / base64). Formalize that with a transport-agnostic AudioSink/AudioSource and a substrate stops caring whether it is LiveKit, raw WebRTC, or a file.

Interface 2 - externalize the persona to character.json

System prompt, tools, mood map, spark palette, and GLB path are hardcoded constants in app.js, duplicated for Dreamfinder and Lyra. That duplication already proves the pattern - it is just not extracted. Make a new character a data file, not a code fork. Same idea as the brain's "feature as config."

The rename: name the Brain API

The two halves share zero code. They communicate by the body sending a line of text to the brain over HTTP and getting back text (plus a mood). That accidental boundary is the best-designed interface in the whole system - it is small, stateless, and already lets you mix any brain with any body.

It just does not have a name or a spec. Write one: POST /brain, { text, context } → { text, mood }. The moment it is a named contract, "my brain + River's body", "River's brain + my avatar", and "both replaced" all become supported configurations instead of happy accidents.

If Andy is building his own bot now, he can use every seam as-is; this page is the map of where the rough edges are, and the best first contributions if he wants to push the engine forward.
