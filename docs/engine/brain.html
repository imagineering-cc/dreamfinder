🧠 The Brain · Dart chat-agent engine

The Brain

Message in → Claude call with tools → reply out. The loop does not know what platform it is on, which LLM it is calling, or what tools exist. That ignorance is the whole point - it is what makes it an engine.

Everything that matters lives in three files under lib/src/agent/. If you delete every other directory and keep these, you still have a working agent core:

FileClassJob
agent_loop.dartAgentLoopThe finite-state loop: call the model, run any tools it asks for, loop, return the final text.
tool_registry.dartToolRegistryWhere tools are registered and looked up. Merges your custom tools with MCP-server tools into one list.
conversation_history.dartConversationHistoryPer-chat memory, stored and replayed as atomic turns so tool-call blocks never get orphaned.

The three seams you plug into

The engine is defined by three injection points. Implement these and the rest is yours.

Seam 1 - the LLM. AgentLoop never imports the Anthropic SDK. It takes a function: 
typedef CreateMessageFn = Future<AgentResponse> Function(
  List<AgentMessage> messages,
  List<ToolDefinition> tools,
  String systemPrompt,
);
Give it one that calls Claude, GPT, a local model, whatever. The loop only cares that it gets back an AgentResponse (text blocks + tool-use blocks + a stop reason). In River this is wired to Claude on the Max plan via direct-Bearer OAuth - see the auth section below.
Seam 2 - the tools. A tool is a name, a JSON-Schema input, and a handler. Register as many as you like: 
toolRegistry.registerCustomTool(CustomToolDef(
  name: 'open_door',
  description: 'Open the named door',
  inputSchema: {'type': 'object', 'properties': {'door': {'type': 'string'}}},
  requiresAdmin: false,
  handler: (args) async => doorService.open(args['door'] as String),
));
The loop calls your handler when the model asks for the tool, feeds the result back, and continues. You can also point the registry at MCP servers (next section) to get whole tool suites without writing Dart.
Seam 3 - the platform. The loop's entry point is platform-neutral: 
Future<String> processMessage(AgentInput input, {required String systemPrompt});

// AgentInput carries only neutral facts about a message:
AgentInput(text: ..., chatId: ..., senderId: ..., isAdmin: ..., isGroup: ...);
Matrix, Discord, Slack, a webhook, a CLI - anything that can produce an AgentInput and deliver the returned string can drive the engine. River's Matrix client (lib/src/matrix/matrix_client.dart) is just one implementation; swap it and the loop is untouched.

Tools without code: MCP

The fastest way to add capability is not to write Dart at all. Drop a server into mcp-config.json:

[
  { "name": "kan", "command": "node", "args": ["mcp-servers/kan/index.js"],
    "env": { "KAN_API_KEY": "$KAN_API_KEY" } }
]

McpManager (lib/src/mcp/mcp_manager.dart) spawns each server, discovers its tools, and the registry merges them in. Adding a capability becomes: write (or grab) an MCP server, add a line, restart. This is how River talks to Kan.bn and Outline.

A minimal bot, end to end

Strip away River's features and the skeleton of any bot on this engine is small:

// 1. The pieces
final history  = ConversationHistory(maxMessages: 40);
final registry = ToolRegistry()..registerCustomTool(myTool);
final loop = AgentLoop(
  createMessage: myClaudeCall,   // Seam 1
  toolRegistry: registry,        // Seam 2
  history: history,
);

// 2. The platform glue (pseudo): for each inbound message...
await for (final msg in myPlatform.incoming()) {
  final reply = await loop.processMessage(            // Seam 3
    AgentInput(text: msg.body, chatId: msg.room,
               senderId: msg.sender, isAdmin: false),
    systemPrompt: 'You are Spark, a helpful assistant.',
  );
  await myPlatform.send(msg.room, reply);
}

That is the whole contract. River's bin/dreamfinder.dart is 1090 lines, but almost all of it is feature wiring (kickstart, dream cycle, standup cron, repo radar, RAG memory) layered on top of exactly this skeleton.

How to build your own bot on it

Step 1 - Implement the LLM seam

Write a CreateMessageFn that calls your model and maps its response into AgentResponse. Reuse River's Anthropic wrapper in bin/dreamfinder.dart if you are using Claude - it already handles OAuth, retries, and error classification.

Step 2 - Bring a platform client

Reuse MatrixClient if you are on Matrix, or write a thin client for your platform exposing roughly: receive events, send a message, join a room, send typing. Translate each inbound event into an AgentInput.

Step 3 - Register your tools and/or MCP servers

Delete River's tool registrations (everything in lib/src/tools/ is River-specific) and register your own. Or wire in MCP servers via the config file.

Step 4 - Write your system prompt

River assembles its prompt from identity + memories + calendar + state in lib/src/agent/system_prompt.dart. Yours can start as a single string passed to processMessage and grow from there.

Step 5 - Persist (optional)

Pass a MessageRepository (SQLite, lib/src/db/) into ConversationHistory and history survives restarts. Skip it and the bot is stateless across reboots. Everything else works the same.

Auth: always Max, never metered

A hard rule in this project: programmatic Claude calls run on the Max plan at zero marginal cost, never on a metered API key. River authenticates with a Claude Code OAuth token (CLAUDE_CODE_OAUTH_TOKEN from claude setup-token) used as a direct Bearer header (anthropic-beta: oauth-2025-04-20). Forced tool use works fine over this path. If you fork the engine, keep this discipline - a metered key that hits a credit wall makes the bot go silently brain-dead, which is exactly the failure this project was hardened against.
